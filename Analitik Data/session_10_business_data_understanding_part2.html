Slide 2

What is Data Understanding?

Data Understanding: The process of exploring, profiling, and assessing data to determine its suitability for analytics objectives.

Why Data Understanding Matters

❌ Without Data Understanding

  • Garbage In, Garbage Out (GIGO)
  • Wrong insights from bad data
  • Wasted modeling effort
  • Missed data quality issues
  • Unrealistic expectations

✓ With Data Understanding

  • Know data limitations
  • Identify data issues early
  • Plan data preparation steps
  • Discover unexpected patterns
  • Set realistic expectations

Data Understanding in Analytics Lifecycle

Data Understanding is part of the Discovery Phase in the Data Analytics Lifecycle

  1. Business Understanding (What problem?)
  2. Data Understanding (What data?) ← We are here
  3. Data Preparation (Clean & transform)
  4. Model Planning (Select algorithms)
  5. Model Building (Train models)
  6. Model Evaluation & Deployment

Real-World Example:

Scenario: A company wants to predict customer churn

Business Understanding: Reduce churn from 25% to 15%

Data Understanding Questions:

  • Do we have historical churn data?
  • How far back does the data go?
  • What customer features are available?
  • Is the data complete and accurate?
  • Are there enough churned customers for training?
  • What's the data refresh frequency?

Time spent on Data Understanding

of total analytics project time

Slide 3

Data Understanding Framework

5 Key Activities

1. Initial Data Collection

Goal: Gather all potentially relevant data sources

  • Identify data sources (databases, files, APIs, external data)
  • Assess data availability and access permissions
  • Document data lineage and ownership
  • Understand data generation process

2. Data Description

Goal: Understand the structure and format of data

  • Number of records (rows)
  • Number of attributes (columns)
  • Data types (numeric, categorical, datetime, text)
  • File formats and sizes
  • Relationships between tables

3. Data Exploration (EDA)

Goal: Discover patterns, distributions, and relationships

  • Descriptive statistics (mean, median, std, min, max)
  • Distribution analysis (histograms, box plots)
  • Correlation analysis
  • Outlier detection
  • Visual exploration

4. Data Quality Assessment

Goal: Evaluate data fitness for purpose

  • Completeness (missing values)
  • Accuracy (correctness)
  • Consistency (no contradictions)
  • Timeliness (up-to-date)
  • Validity (follows business rules)
  • Uniqueness (no duplicates)

5. Data Profiling Report

Goal: Document findings and recommendations

  • Summary of data sources
  • Data quality scores
  • Key findings and insights
  • Identified issues and risks
  • Recommendations for data preparation

Slide 5

Data Quality Dimensions

Data Quality: The degree to which data meets requirements for its intended use.

6 Critical Dimensions of Data Quality

1. Completeness

Extent to which data is not missing

$$\text{Completeness} = \frac{\text{Non-null values}}{\text{Total values}} \times 100\%$$

Example: Dataset has 1000 rows, "email" column has 850 non-null values

$$\text{Completeness} = \frac{850}{1000} \times 100\% = 85\%$$

Medium Quality

2. Accuracy

Data correctly represents real-world values

$$\text{Accuracy} = \frac{\text{Correct values}}{\text{Total values}} \times 100\%$$

Example: Verify 100 phone numbers, 92 are valid

$$\text{Accuracy} = \frac{92}{100} \times 100\% = 92\%$$

High Quality

3. Consistency

Same data across different sources matches

Example Issues:

  • Customer name in CRM: "John Smith" vs Billing: "J. Smith"
  • Date format: "2025-01-15" vs "15/01/2025"
  • Units: "100 kg" vs "220 lbs" (same weight, different units)

4. Timeliness

Data is up-to-date and available when needed

Questions:

  • When was data last updated?
  • Is the data fresh enough for decision-making?
  • What's the acceptable data age?

5. Validity

Data conforms to defined business rules and constraints

Example Rules:

  • Age must be between 0-150
  • Email must contain "@"
  • Phone number has 10-15 digits
  • Date of birth < Today

$$\text{Validity} = \frac{\text{Valid values}}{\text{Total values}} \times 100\%$$

6. Uniqueness

No unintended duplicate records

$$\text{Uniqueness} = \frac{\text{Unique records}}{\text{Total records}} \times 100\%$$

Example: 10,000 customer records, 9,800 unique customers (200 duplicates)

$$\text{Uniqueness} = \frac{9800}{10000} \times 100\% = 98\%$$

High Quality
Slide 6

Data Quality Assessment

Quality Score Calculation

Overall Data Quality Score:

$$\text{Quality Score} = \frac{\sum_{i=1}^{n} w_i \times Q_i}{\sum_{i=1}^{n} w_i}$$

Where:

  • \(Q_i\) = Quality score for dimension \(i\)
  • \(w_i\) = Weight/importance of dimension \(i\)
  • \(n\) = Number of quality dimensions

Example: Customer Dataset Quality Assessment

Dimension Weight Score (%) Weighted Score Status
Completeness 25% 85% 21.25 Medium
Accuracy 30% 92% 27.60 High
Consistency 15% 78% 11.70 Medium
Validity 20% 95% 19.00 High
Uniqueness 10% 98% 9.80 High
Overall Quality Score 89.35% High

Calculation:

$$\text{Quality Score} = \frac{(0.25 \times 85) + (0.30 \times 92) + (0.15 \times 78) + (0.20 \times 95) + (0.10 \times 98)}{1.0}$$

$$= \frac{21.25 + 27.60 + 11.70 + 19.00 + 9.80}{1.0} = 89.35\%$$

Quality Score Interpretation

High Quality

≥ 85%

Data is fit for purpose. Minor issues can be addressed in data preparation.

Medium Quality

70-84%

Significant data preparation required. May limit analytics scope.

Low Quality

< 70%

Major issues. Consider finding alternative data sources or improving data collection.

Exploratory Data Analysis (EDA)

EDA: The process of analyzing datasets to summarize their main characteristics, often using statistical graphics and other visualization methods.

EDA Objectives

  • Understand data structure and distributions
  • Detect outliers and anomalies
  • Discover patterns and relationships
  • Test underlying assumptions
  • Generate hypotheses for analytics

Descriptive Statistics

Central Tendency

Mean (Average):

$$\bar{x} = \frac{1}{n} \sum_{i=1}^{n} x_i$$

Median:

Middle value when sorted

Mode:

Most frequent value

Dispersion (Spread)

Variance:

$$\sigma^2 = \frac{1}{n} \sum_{i=1}^{n} (x_i - \bar{x})^2$$

Standard Deviation:

$$\sigma = \sqrt{\sigma^2}$$

Range:

$$\text{Range} = x_{max} - x_{min}$$

Example: Customer Age Distribution

Statistical Summary:

Metric Value Interpretation
Count 10,000 Total customers
Mean 42.5 years Average customer age
Median 40 years Middle value (50th percentile)
Std Dev 12.8 years Typical deviation from mean
Min 18 years Youngest customer
Max 85 years Oldest customer
Q1 (25%) 32 years 25% of customers are younger
Q3 (75%) 53 years 75% of customers are younger
Slide 8

Correlation Analysis

Pearson Correlation Coefficient

$$r = \frac{\sum_{i=1}^{n} (x_i - \bar{x})(y_i - \bar{y})}{\sqrt{\sum_{i=1}^{n} (x_i - \bar{x})^2} \sqrt{\sum_{i=1}^{n} (y_i - \bar{y})^2}}$$

Range: \(-1 \leq r \leq 1\)

  • \(r = 1\): Perfect positive correlation
  • \(r = 0\): No correlation
  • \(r = -1\): Perfect negative correlation

Correlation Strength Interpretation

|r| Value Strength Interpretation
0.00 - 0.19 Very Weak Almost no relationship
0.20 - 0.39 Weak Slight relationship
0.40 - 0.59 Moderate Moderate relationship
0.60 - 0.79 Strong Strong relationship
0.80 - 1.00 Very Strong Very strong relationship

Example: Customer Churn Correlations

Key Findings:

  • Strong positive: Support_Tickets ↔ Churn (r = 0.72) - More support issues → Higher churn
  • Strong negative: Tenure ↔ Churn (r = -0.68) - Longer tenure → Lower churn
  • Moderate positive: Monthly_Charges ↔ Churn (r = 0.45) - Higher charges → Higher churn
  • Weak: Age ↔ Churn (r = 0.12) - Age has minimal impact on churn

⚠️ Important: Correlation ≠ Causation

Just because two variables are correlated doesn't mean one causes the other!

Example: Ice cream sales and drowning deaths are correlated (both higher in summer), but ice cream doesn't cause drowning!

Slide 9

Outlier Detection

Outlier: A data point that differs significantly from other observations. May indicate data error or genuine extreme value.

Method 1: IQR (Interquartile Range) Method

Formulas:

$$\text{IQR} = Q_3 - Q_1$$

$$\text{Lower Bound} = Q_1 - 1.5 \times \text{IQR}$$

$$\text{Upper Bound} = Q_3 + 1.5 \times \text{IQR}$$

Outliers: Values below Lower Bound or above Upper Bound

Example: Monthly Purchase Amount

Given:

  • Q1 = $50
  • Q3 = $200

Calculate:

$$\text{IQR} = 200 - 50 = 150$$

$$\text{Lower Bound} = 50 - 1.5 \times 150 = 50 - 225 = -175$$

$$\text{Upper Bound} = 200 + 1.5 \times 150 = 200 + 225 = 425$$

Interpretation: Purchases above $425 are outliers (unusually high)

Method 2: Z-Score Method

Formula:

$$z = \frac{x - \mu}{\sigma}$$

Where:

  • \(x\) = data point
  • \(\mu\) = mean
  • \(\sigma\) = standard deviation

Rule of thumb: |z| > 3 indicates an outlier

Example: Customer Age

Given: Mean age = 42 years, Std Dev = 12 years, Customer age = 85 years

Calculate:

$$z = \frac{85 - 42}{12} = \frac{43}{12} = 3.58$$

Interpretation: |z| = 3.58 > 3, this is an outlier (unusually old customer)

Box Plot Visualization

✓ When to Keep Outliers

  • Legitimate extreme values
  • VIP customers with high spending
  • Rare but valid events
  • Business wants to understand extremes

✗ When to Remove Outliers

  • Data entry errors
  • Measurement errors
  • System glitches
  • Will skew model training
Slide 10

Missing Data Analysis

Types of Missing Data

1. MCAR - Missing Completely At Random

Missingness has no relationship with any data (observed or unobserved)

Example: Survey responses lost due to server crash

Impact: Least problematic, can usually be ignored or imputed

2. MAR - Missing At Random

Missingness depends on observed data, not on the missing value itself

Example: Younger customers less likely to report income (age is observed, income is missing)

Impact: Can be handled with appropriate imputation methods

3. MNAR - Missing Not At Random

Missingness depends on the unobserved value itself

Example: High-income people refuse to report income (missing because value is high)

Impact: Most problematic, may introduce bias

Missing Data Patterns

Handling Missing Data

Method When to Use Pros Cons
Deletion < 5% missing, MCAR Simple, no bias if MCAR Loses data, reduces sample size
Mean/Median Imputation Numerical data, < 15% missing Simple, fast Reduces variance, doesn't account for relationships
Mode Imputation Categorical data Simple, preserves categories May bias toward majority class
Forward/Backward Fill Time series data Maintains temporal continuity Only for time-ordered data
Predictive Imputation MAR, complex patterns Accounts for relationships, more accurate Computationally expensive, complex
Create "Missing" Category MNAR, informative missingness Preserves information about missingness Increases dimensionality

Decision Framework: What % Missing is Too Much?

  • 0-5%: Usually safe to delete or impute
  • 5-15%: Use careful imputation methods
  • 15-40%: Investigate pattern, consider creating "missing" indicator
  • > 40%: Consider dropping the variable entirely or collecting more data

  • Data quality problems
  • Outliers and anomalies

  • Potential impact on analytics

  • Feature engineering opportunities
  • Additional data needed
  • Data collection improvements

Summary & Best Practices

Business & Data Understanding: The Foundation

Remember:

  • Business Understanding defines WHAT problem to solve
  • Data Understanding determines IF we CAN solve it
  • Together, they form the Discovery Phase

Key Takeaways

Business Understanding

  • ✓ Use SMART objectives
  • ✓ Identify all stakeholders
  • ✓ Define clear problem statement
  • ✓ Translate to analytics problem
  • ✓ Set measurable success criteria
  • ✓ Estimate business impact

Data Understanding

  • ✓ Assess all data sources
  • ✓ Evaluate data quality (6 dimensions)
  • ✓ Perform thorough EDA
  • ✓ Detect outliers and missing data
  • ✓ Analyze correlations
  • ✓ Document findings in report

Common Pitfalls to Avoid

❌ Don't do this:

  • Skip business understanding and jump to modeling
  • Assume data quality is good without checking
  • Ignore stakeholder input
  • Define vague, unmeasurable objectives
  • Proceed with low-quality data (< 70%)
  • Forget to document assumptions and findings
  • Overlook missing data patterns

Best Practices Checklist

✓ Before Moving to Data Preparation:

  • ☐ Business objective is SMART
  • ☐ All stakeholders identified and consulted
  • ☐ Problem statement is clear and specific
  • ☐ Success metrics defined and agreed upon
  • ☐ Analytics approach selected (descriptive/predictive/prescriptive)
  • ☐ Data sources identified and accessible
  • ☐ Data quality score ≥ 70% (preferably ≥ 85%)
  • ☐ EDA completed and documented
  • ☐ Missing data strategy defined
  • ☐ Outlier handling approach determined
  • ☐ Data profiling report created and approved
  • ☐ Go/No-go decision made
